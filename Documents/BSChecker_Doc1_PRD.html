3. Bot Rules & Behavioural Logic

3.1 Core Rules (Non-Negotiable)

These rules govern every interaction Nah That's Fake has with users. They cannot be overridden by any configuration.

  1. Rule 1 â€” Always express results as probability, never as fact.

The bot never says 'This is fake.' It always says '87% likely to be AI-generated.' This protects us legally and is more accurate.

  1. Rule 2 â€” Always include a disclaimer on every result.

Every response ends with: 'Nah That's Fake provides probabilistic analysis only. It is not a legal determination. Use your own judgement.'

  1. Rule 3 â€” Never store the actual content of checked media.

We store metadata (hash, timestamp, user ID, result score) but never the image file itself. Links are stored in hashed form only. This is critical for privacy compliance.

  1. Rule 4 â€” Rate limits are enforced per user, not per device.

A user's daily free quota of 3 checks applies to their Telegram account ID. They cannot bypass it by using a different device.

  1. Rule 5 â€” Groups get a shared pool, not per-member limits.

In a free group, all members share 10 checks per day total. In a premium group, checks are unlimited for all members.

  1. Rule 6 â€” Referral credits are only awarded when the referred user completes their first check.

Simply signing up does not trigger the referral reward. The referee must perform at least one scan. This prevents credit farming.

  1. Rule 7 â€” The bot must always reply publicly in groups when tagged.

This is the viral mechanism. When tagged in a group, the bot responds in the group thread, not via DM. The response includes a subtle self-promotion line.

  1. Rule 8 â€” Abuse detection: flag accounts with >50 checks in 1 hour for manual review.

This catches bot abuse and API cost attacks. Flagged accounts are rate-limited to 1 check per 10 minutes until reviewed.

4. Detection Logic â€” Phase 1

4.1 Image Detection Flow

When a user sends an image to the bot, the following sequence executes:

  1. Receive image from Telegram API as file_id.
  2. Download image file to temporary memory (never to permanent disk).
  3. Generate SHA-256 hash of the image file.
  4. Check hash against our local cache â€” if we've seen this exact image before, return cached result instantly (saves API calls).
  5. If not cached: send image to Sightengine /check endpoint with models=type,genai.
  6. Receive probability score (0.0â€“1.0) from Sightengine.
  7. Map score to Risk Level: >0.80 = HIGH, 0.40â€“0.80 = MEDIUM, <0.40 = LOW.
  8. Store hash + score + timestamp + user_id in Supabase (not the image).
  9. Delete image from memory.
  10. Return formatted response to user.

ğŸ’¡ Caching Strategy

If the same image hash has been checked before, we return the cached result without calling Sightengine.

This dramatically reduces API usage as viral fake images get checked by many users.

Cache TTL: 30 days. After 30 days, re-check in case detection models have improved.

4.2 Link Detection Flow

When a user sends a URL, the following sequence executes:

  1. Extract and normalise the URL (remove tracking parameters, expand shortened URLs).
  2. Check URL against local cache (hash of normalised URL).
  3. If not cached: simultaneously query Google Safe Browsing API and VirusTotal API.
  4. Google Safe Browsing returns: SAFE or flagged with threat type.
  5. VirusTotal returns: number of security vendors that flagged the URL (out of ~90).
  6. Combine signals: if either flags it, escalate to at least MEDIUM risk.
  7. Risk mapping: 0 flags = LOW, 1â€“2 flags = MEDIUM, 3+ flags = HIGH.
  8. Store normalised URL hash + combined result in Supabase.
  9. Return formatted response.
